# 业务历史解读周报写法模式

本参考来自 `D:\Downloads\监管政策周报_合并完整版_最终版.docx` 的结构化抽取，用于生成正式版时对齐业务口径。它不是脚本生成正文的规则库，而是 Codex 写稿和自检时必须遵守的样稿经验。

重要边界：历史周报是“写法模式库”，不是“句子素材库”。生成新一期时只能抽象学习结构、颗粒度、主体映射和高亮逻辑；不得复用历史周报中已经写好的政策摘要、平安影响段或连续业务判断表达。若历史周报里包含本次同一期或同一政策，只能用于确认业务专家为什么这样分层，不能把原句搬入 `final_report_content.json`。

## 总体结构

每期使用以下版式：

```text
监管政策周报（合并版）
#YYYY.MM.DD
1. 发文机构《政策名称》
P1：政策摘要
P2：平安影响（可为空）
```

条目标题必须是“发文机构 + 《文件名》”。输入可只给文件名，但最终 `items[].title` 要呈现为业务样稿标题口径。

## 反复用要求

每次写稿后都要完成一次“样稿相似度自检”：

- 允许一致：政策标题、机构名称、发布日期、文号、生效日期、征求意见截止日、法律条款名等事实短语。
- 不允许一致：P1 的制度定位句、P1 的要点组织句、P2 的专业公司动作句、benchmark/opportunity 的判断句。
- 如果某段与历史样稿只是替换少量词语，仍视为过拟合，必须改写。
- 改写不是降质：事实不变，重新安排句序、压缩或拆分制度要点，并用本次政策原文重新组织平安影响。
- 高亮片段也要随新句子重新选择，不能沿用历史样稿的高亮清单。

## P1 写法

P1 的职责是交代监管事实，不写平安动作。重点政策通常包含：

- 发布日期和发文机构。
- 文件名称、文号或发文字号。
- 文件性质或监管定位，例如“首部系统性规则”“专项规范”“顶层行动指南”。
- 生效日期、过渡期、征求意见截止日期或当前阶段。
- 4-7 个核心制度变化，常用“包括：……”集中列举。

常见动词：

- 正式规则：发布、印发、联合发布、公布、修订发布。
- 草案/意见稿：公开征求意见、就……征求意见。
- 法律/司法解释：审议通过、作出规定、明确适用口径。
- 专项行动：部署、组织开展、联合开展。

避免写法：

- 不写“对平安有影响”“建议关注”等动作口吻。
- 不用宏观评价替代制度要点。
- 不扩写没有来源支撑的数字、期限、比例和适用范围。

## P2 写法

P2 的职责是把政策翻译成平安内部的受影响主体、业务链路和动作。历史样稿不是“一个主体一个动作”的固定格式，而是按政策影响范围分层。

### direct：直接适用

适用于金融产品网络营销、上市公司治理、董秘规则、资本市场规则、数据合规、保险销售、反洗钱等直接改变合规义务的文件。

写法：

- 可用“《办法》适用于平安集团及旗下各金融子公司：……”开头。
- 多主体并列时，每个主体单独落到业务链路和动作，用分号切分。
- 主体要下钻到产品线、渠道、页面、团队，例如“平安银行数字营销矩阵”“灵活宝和闲钱宝”“壹钱包收银台页面”“平安产险、平安寿险第三方平台合作模式”。
- 不要把多个主体合并成“相关单位”“各专业公司”。

样稿模式：

```text
平安银行需清理数字营销矩阵……；壹钱包需调整收银台页面……；平安产险、平安寿险需全面检视、调整与第三方平台的合作模式……
```

### scene_direct_related：场景直接相关

适用于政策未点名平安，但明确落到已有业务场景，如客户数据出境、海事理赔、农险地块数据、设备融资、信托财产登记。

写法：

- 先说明相关业务场景，再点名主体。
- 重点写“哪条业务链路会变”，而非泛泛写制度映射。
- 可以只写 1-2 个主体，但必须具体。

### opportunity：业务机会或政策导向

适用于宏观产业政策、服务业政策、养老健康政策、科技创新支持政策等。

写法：

- 先界定“非强制义务”：例如“属于宏观产业政策，不直接对金融机构设定强制性义务”。
- 再写机会方向：哪个主体可关注哪些产品线或生态布局。
- 不写整改、排查、立即落实。

样稿模式：

```text
平安产险可关注科技保险、绿色保险产品线；平安健康、平安寿险、平安养老险可检视医疗养老生态布局与政策导向的契合度。
```

### benchmark：对标参考

适用于央企内控、政府治理、行业自律、非金融专属制度但有管理借鉴价值的文件。

写法：

- 明确“不直接适用”或“不直接适用于平安集团及其下属机构”。
- 只写对标、借鉴、研究，不写强整改。
- 重大顶层文件允许长 P2，解释为什么值得对标。

### none：无明显关联

与平安主营或集团职能没有明确关系时只写 P1。不要为了凑两段写空泛影响。

## 受影响主体颗粒度

优先使用历史样稿中真实出现的业务颗粒度：

- 集团层：平安集团、集团合规部、集团科技条线、集团反洗钱团队、集团消保团队。
- 银行/支付：平安银行、平安银行普惠业务、平安银行对公业务、灵活宝、闲钱宝、壹钱包。
- 保险：平安产险、平安产险农险部、平安寿险、平安养老险、平安健康险、平安健康。
- 资管/资本市场：平安证券、平安期货、平安基金、平安资管、平安理财。
- 信托/租赁/消金/平台：平安信托、平安租赁、平安消金、陆控、陆控担保业务（融易）、陆控消金业务。

使用规则：

- 如果政策影响具体产品、页面、渠道、团队，优先写具体颗粒度。
- 如果只影响持牌主体整体，写持牌主体。
- 如果需要集团统筹，写集团职能团队，不把所有专业公司列一遍。

## 高亮逻辑

历史样稿中红色文字全部加粗。渲染前必须在 `final_report_content.json` 中明确填写精确子串：

- `paragraph1_red_bold`
- `paragraph1_black_bold`
- `paragraph2_red_bold`
- `paragraph2_black_bold`

### 红色加粗

用于监管强信号：

- 生效日、过渡期、截止日：如 `2026年9月30日`、`过渡期至2027年12月31日`。
- 核心制度框架：如 `从营销资质、内容规范、行为规范、合作规范四个维度构建了`。
- 硬性禁止或强制要求：如 `不得兼任`、`建立定期评价机制`、`不得存在兼任总经理、分管业务副经理、财务负责人等冲突情形`。
- 关键比例、阈值、适用范围变化：如 `权重不低于50%`、`由0.5提升至1.5`。
- 重要限定：如 `正式公布的官方原文尚未发布。`

### 黑色加粗

用于业务定位：

- 受影响平安主体：如 `平安银行`、`壹钱包`、`平安产险、平安寿险`、`平安集团、平安银行`。
- 内部工作标记：如 `【已做专项解读】`、`【集团反洗钱团队已做专项解读】`、`【已另行形成专项解读】`。
- 不直接适用判断中的关键限定：如 `不直接适用于平安集团及其下属机构`。

### 标注边界

- 不整段标红，不把普通背景句标红。
- P1 主要标监管事实，P2 主要标主体和关键义务。
- 同一句中主体用黑色加粗，硬性要求用红色加粗。
- 如果某个高亮片段不是段落原文中的精确子串，必须先改成精确子串再渲染。

## 4 月 26 日对照结论

业务样稿和旧输出的最大差距集中在三点：

- `金融产品网络营销管理办法` 的 P2 必须按专业公司和产品线拆分，至少覆盖平安银行、灵活宝/闲钱宝、壹钱包、平安产险和平安寿险，不能写成“相关金融子公司需关注网络营销合规”。
- `上市公司董事会秘书监管规则` 的 P2 只直接适用于 A 股上市主体平安集团、平安银行，核心落点是董秘任职冲突、提名委员会审查、履职评价和责任追究，不应扩大到所有金融子公司。
- `推进服务业扩能提质的意见` 属于宏观产业政策，应写“非强制义务 + 业务机会/生态布局契合度”，不写整改。

最终 Word 的落差也很明确：旧输出只有少量红字，且条目标题不是 `Heading 3`；业务样稿有稳定的红色加粗和黑色主体加粗组合。因此正式出稿必须先在 JSON 中显式标注高亮片段，再渲染 Word。
